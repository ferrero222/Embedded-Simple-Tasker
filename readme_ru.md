## Обзор

**Simple Tasker** — это легковесное, платформенно-независимое, кооперативное ядро реального времени (RTC). Оно реализует парадигму Активных Объектов (Active Objects). Создано для систем НЕ жесткого реального времени с ограниченным числом ресурсов или ограниченным доступоv к железу, то когда нет возможности использовать классические RTOS.

### Ключевые особенности:
- **Нулевое динамическое выделение памяти:** Никаких `malloc`/`free`. Вся память (очереди, пулы подписок, таймеры) выделяется пользователем статически.
- **Гибкая система событий без глобального пула:** Поддержка обычных и mutable событий, которые хранятся напрямую в очереди задачи.
- **Встроенные таймеры:** Поддержка периодических и однократных (one-shot) таймеров.
- **Механизм Pub/Sub:** Слабосвязанная коммуникация между задачами через сигналы.
- **Приоритетное планирование:** Задачи с более высоким приоритетом всегда обрабатываются первыми.

## Быстрый старт

```c
#include "simple_tasker.h"
#include <stdio.h>

// 1. Объявляем задачу и её очередь (размер степень двойки)
static st_task_t my_task;
static st_event_t my_queue[16]; 

// 2. Пишем обработчик событий
static void my_handler(st_task_t *me, st_event_t const *e) {
    if (e->sig == ST_SIGNAL_INIT) {
        printf("Task initialized!\n");
    } else if (e->sig == ST_SIGNAL_USER) {
        printf("User event received!\n");
    }
}

int main(void) {
    // 3. Инициализируем ядро
    st_init(NULL); 
    
    // 4. Конструируем и запускаем задачу
    st_task_ctor(&my_task, 1, my_handler, my_queue, 16);
    st_task_start(&my_task); 
    
    // 5. Главный цикл
    while (1) {
        st_tick();   // Вызывать из прерывания системного таймера (например, 1 мс)
        st_run();    // Кооперативный цикл обработки событий
    }
}
```

## Руководство пользователя

### 1. Справочник API

#### Инициализация и задачи
* **`void st_init(st_idle_t fn_idle)`**  
  Инициализирует внутренние структуры ядра. `fn_idle` — функция, которая будет вызываться, когда ни у одной задачи нет событий (идеально для перевода MCU в режим сна, например, `__WFI()`).

* **`void st_task_ctor(st_task_t *me, uint8_t prio, st_handler_t handler, st_event_t *queue, uint8_t qsize)`**  
  Регистрирует задачу в планировщике.  
  **ВАЖНО:** `qsize` (размер очереди в чанках) **обязан быть степенью двойки** (2, 4, 8, 16, 32, 64, 128, 256). Максимальное значение — 255.

* **`void st_task_start(st_task_t *me)`**  
  Запускает задачу, синхронно вызывая её обработчик с событием `ST_SIGNAL_INIT`. Используется для начальной настройки задачи (запуск таймеров, подписка на сигналы).

#### Отправка событий
* **`bool st_post(st_task_t *me, st_event_t const *e)`**  
  Отправляет **обычное** событие (занимает ровно 1 чанк) в очередь конкретной задачи. Возвращает `false`, если очередь переполнена.

* **`bool st_post_mutable(st_task_t *me, void const *e, uint8_t size)`**  
  Отправляет **mutable** событие с полезной нагрузкой. Ядро автоматически вычисляет необходимое количество памяти из буфера событий задачи.  
  **ВАЖНО:** Структура события `e` **обязана** начинаться с поля типа `st_event_t` (или иметь идентичную компоновку первых 4 байт).

#### Pub/Sub (Издатель-Подписчик)
* **`void st_subscribe(st_signal_t sig, st_task_t *me)`**  
  Подписывает задачу `me` на получение событий с сигналом `sig`.

* **`void st_unsubscribe(st_signal_t sig, st_task_t *me)`**  
  Отписывает задачу. Освобождает внутренний узел подписки для повторного использования.

* **`void st_publish(st_event_t const *e)`**  
  Отправляет копию события `e` всем задачам, подписанным на `e->sig`. Работает только с обычными (1 чанк) событиями.

#### Таймеры
* **`void st_timer_start(st_timer_t *tmr, st_task_t *task, st_signal_t sig, uint32_t interval, uint32_t counter)`**  
  Запускает таймер. `counter` — начальная задержка в тиках (>0). `interval` — период повторения (0 = однократный срабатывание).  
  **ВАЖНО:** Безопасно вызывать повторный старт для уже работающего таймера (ядро автоматически удалит старую запись из списка).

* **`void st_timer_stop(st_timer_t *tmr)`**  
  Останавливает таймер и удаляет его из внутреннего списка для экономии процессорного времени.

* **`void st_tick(void)`**  
  Декрементирует счетчики всех активных таймеров. Должна вызываться периодически (например, из прерывания SysTick каждые 1 мс).

#### Исполнение
* **`void st_run(void)`**  
  Бесконечный кооперативный цикл. Находит задачу с наивысшим приоритетом, имеющую события, извлекает событие и передает его обработчику. Если событий нет, вызывает `fn_idle`.

### 2. Критические нюансы для пользователя

#### А. Выравнивание памяти (Memory Alignment)
Структура `st_event_t` имеет размер 4 байта. Если вы создаете mutable-события, содержащие 32-битные поля (`uint32_t`, `float`, указатели), **массив очереди задачи должен быть выровнен по границе 4 байт**. На архитектурах вроде RISC-V несовместимое выравнивание приведет к `HardFault` при попытке обработчика прочитать данные.
```c
// Правильно (GCC / ARM):
__attribute__((aligned(4))) static st_event_t my_queue[32];

// Правильно (C11):
_Alignas(4) static st_event_t my_queue[32];
```

#### Б. Безопасность при вызове из прерываний (ISR Safety)
Функции `st_post`, `st_post_mutable`, `st_publish` и `st_tick` **не являются потокобезопасными по умолчанию**. Они не используют мьютексы, чтобы сохранить минимальный оверхед.  
Если вы вызываете их из обработчика прерывания (ISR), вы **обязаны** определить макросы обработчки критических секций:
```c
ST_ENTER_CRITICAL(); // Например: __disable_irq();
ST_EXIT_CRITICAL();  // Например: __enable_irq();
```
Внутри функций макросы уже прописаны.

#### В. Ограничения Mutable-событий (Strict Boundaries)
Очередь событий для задачи делится на чанки соразмерные размеру структуры ивента. Если мы постим обычный ивент, используется только один чанк.
Когда мы хотим запостить mutable событие то в зависимсоти от размера структуры такого события будет выделено соответствующее число чанков необходимого суммарного размера. 
Вообщем нужно следить чтобы mutable структура события по размеру не превышала оставшееся место в очереди событий задачи иначе она не добавится.

## Потребление памяти (Memory Footprint)

Simple Tasker спроектирован с учетом жестких ограничений встраиваемых систем. Ниже приведена оценка потребления памяти для 32-битной архитектуры (например, ARM Cortex-M, где размер указателя равен 4 байтам). 

#### 1. Статический оверхед ядра (RAM)
Эта память резервируется ядром автоматически при компиляции, независимо от количества созданных задач.

| Структура данных | Размер | Описание |
| :--- | :--- | :--- |
| `st_task_reg` | 64 байта | Массив указателей на задачи 16(ST_MAX_TASKS) (16 × 4 байта) |
| `st_sub_pool` | 800 байт | Статический пул узлов 100(ST_MAX_SUBS) подписок (100 × 8 байт) |
| `st_sub_list` | 256 байт | Массив 64(ST_MAX_SIGNALS) голов списков подписок (64 × 4 байта) |
| `st_tmr_pool` | 256 байт | Статический пул таймеров 16(ST_MAX_TIMERS)(16 × 16 байт с учетом выравнивания) |
| Служебные переменные | ~20 байт | Счетчики, указатели `head` списков, флаги |
| **ИТОГО (Ядро)** | **~1.4 КБ** | **Фиксированное потребление ОЗУ ядром при такой настройке** |

#### 2. Потребление на пользователя (RAM)
Эту память вы выделяете самостоятельно при объявлении задач и таймеров.

| Объект | Размер на 1 экземпляр | Примечание |
| :--- | :--- | :--- |
| Структура `st_task_t` | 12 байт | Сама задача (без учета очереди) |
| Структура `st_timer_t` | 16 байт | Экземпляр таймера |
| Очередь задачи | `N × 4` байта | Где `N` — размер очереди (`qsize`). 1 чанк = 4 байта. |
| **Пример задачи** | **~76 байт** | Задача + очередь на 16 событий (12 + 16×4) |

#### 3. Размер кода (ROM / Flash)
При компиляции с оптимизацией по размеру (`-Os` в GCC/Clang):
* **Базовое ядро:** ~1.5 – 2.0 КБ.
* Если вы не используете таймеры или Pub/Sub, компоновщик (linker) с флагом `-ffunction-sections` и `-fdata-sections` (и последующим `--gc-sections`) автоматически вырежет неиспользуемые функции, уменьшив размер до **~1.0 – 1.2 КБ**.

#### 4. Как уменьшить потребление памяти (Масштабирование)
Если 1.4 КБ статического ОЗУ слишком много для вашего чипа, вы можете легко уменьшить его, изменив макросы в начале файла `simple_tasker.h` под ваши реальные нужды:

```c
// Значения по умолчанию:
#define ST_MAX_TASKS         16U   // Уменьшите до 4-8, если задач мало (-48 байт ОЗУ)
#define ST_MAX_SIGNALS       64U   // Уменьшите до 16-32, если сигналов мало (-128 байт ОЗУ)
#define ST_MAX_SUBS          100U  // Уменьшите до 20-30, если подписок мало (-560 байт ОЗУ!)
#define ST_MAX_TIMERS        16U   // Уменьшите до 4-8, если таймеров мало (-128 байт ОЗУ)
```
*Пример:* При установке `ST_MAX_SUBS = 30` и `ST_MAX_TIMERS = 8`, статический оверхед ядра снижается до **~600 байт**, что делает планировщик пригодным даже для маломощных MCU с 2-4 КБ ОЗУ.

## Руководство разработчика (Внутренняя архитектура)

Этот раздел описывает внутреннее устройство ядра для тех, кто планирует его модифицировать или портировать.

### 1. Управление памятью событий (Chunking System)
В отличие от классических реализаций с глобальным пулом событий (event pool), Simple Tasker хранит события **напрямую в плоском массиве очереди задачи**.
* **Структура чанка:** Очередь — это массив `st_event_t`. Каждое событие начинается с 4-байтового заголовка:
  ```c
  typedef struct st_event_t {
      st_signal_t sig;      // Сигнал
      uint8_t     slots;    // Количество смежных чанков, занимаемых событием (>= 1)
      uint8_t     _reserved[2]; // Выравнивание до 4 байт
  } st_event_t;
  ```
* **Механика:** `st_post_mutable` вычисляет нужное количество чанков через макрос `ST_EVENT_CHUNKS(size)` и выполняет `memcpy` данных пользователя напрямую в `me->queue[me->head]`. Поле `slots` принудительно перезаписывается ядром для гарантии корректного освобождения памяти.
* **Дефрагментация:** При извлечении события `tail` сдвигается на `e->slots`. Если после этого `nused == 0`, `head` и `tail` сбрасываются в `0`, устраняя любую фрагментацию.

### 2. Механизм подписок (Pub/Sub Internals)
Подписки реализованы с защитой от фрагментации и утечек памяти, без использования `malloc`.
* **Массив голов:** `st_sub_list[ST_MAX_SIGNALS]` хранит указатели на головы односвязных списков для каждого сигнала.
* **Статический пул + Free List:** Используется массив `st_sub_pool[ST_MAX_SUBS]` и указатель `st_sub_free_list`.
  * При `st_subscribe`: Узел берется из головы `st_sub_free_list` (O(1)). Если список пуст, берется новый узел из `st_sub_pool`. Узел добавляется в голову списка `st_sub_list[sig]`.
  * При `st_unsubscribe`: Узел находится в списке, выцепляется из него и возвращается в голову `st_sub_free_list` (O(1)). Это **полностью предотвращает исчерпание пула** при частых подписках/отписках.

### 3. Механика таймеров
* Таймеры хранятся в односвязном списке `st_tmr_head`.
* **Защита от дублирования:** Функция `st_timer_start` явно ищет таймер в списке и удаляет его перед добавлением. Это предотвращает создание циклических ссылок или дубликатов при повторном вызове `st_timer_start` для one-shot таймера.
* **Оптимизация CPU:** При срабатывании one-shot таймера (`interval == 0`), он автоматически удаляется из списка `st_tmr_head` внутри `st_tick()`, чтобы в будущем не тратить такты процессора на проверку неактивных таймеров.

### 4. Планировщик (Scheduler)
* Задачи регистрируются в массиве `st_task_reg` и **автоматически сортируются по убыванию приоритета** (Insertion Sort) при вызове `st_task_ctor`.
* Функция `st_sched()` просто итерирует этот массив сверху вниз и возвращает первую задачу, у которой `nused > 0`. Это гарантирует строгое приоритетное выполнение без сложных битовых масок готовности, что экономит ROM и упрощает код.
